# Hot Sheet — AI Requirements Summary

**Purpose.** A single-file synthesis of every requirements document under `docs/`. Read this to understand *what Hot Sheet is supposed to do* without reading each doc front-to-back. Pair it with `docs/ai/code-summary.md` (*where the code lives*).

**Status legend.** Each entry is tagged:
- **Shipped** — fully implemented and tested.
- **Partial** — core shipped, enumerated gaps noted.
- **Design only** — doc exists, implementation not yet built.
- **Deferred** — intentionally postponed; see the entry for the trigger to revisit.

**Maintenance.** See §15. Update this file in the same change as any requirements doc change.

---

## 1. Product vision & foundational decisions

Hot Sheet is a local-first, developer-focused ticket tracker that lives next to a codebase. A CLI (`hotsheet`) spins up an embedded HTTP server + browser UI; data lives in `.hotsheet/` beside the project; a Tauri wrapper offers a native desktop experience. The product's central bet is **AI-integration**: tickets are auto-exported to markdown worklists, and an MCP channel lets Claude Code work through the worklist turn-by-turn.

- **Non-functional requirements** (§1): strict data locality, debounces tuned for responsiveness (worklist 500 ms, open-tickets 5 s), parameterized SQL only, soft-delete everywhere, 3-tier auto backups, signed desktop updates, cross-platform (macOS, Linux, Windows).
- **Build outputs** (§1): `dist/cli.js`, `dist/channel.js`, `dist/client/app.global.js`, `dist/client/styles.css`.
- **Status:** Shipped.

## 2. Data & storage (`2-data-storage.md`)

PGLite embedded Postgres with automatic additive migrations, tables for tickets (HS-prefixed auto-increment IDs, never reused), attachments, stats snapshots, command log, settings (plugin-only now — project config moved to `.hotsheet/settings.json`), plus sync tables (`ticket_sync`, `sync_outbox`, `note_sync`). Notes stored as JSON array `{ id, text, created_at }`. File-based settings.json carries `appName`, `backupDir`, `appIcon`, `ticketPrefix`, `secret`, `port`, UI layout keys. A `.hotsheet/hotsheet.lock` file (pid/startedAt) enforces single-instance per data dir; `~/.hotsheet/instance.json` + `projects.json` support multi-project.

**Status:** Shipped (full schema, migrations, stale-lock cleanup, multi-project registry). Cross-refs: §3 (notes), §7 (backupDir), §18 (plugin_enabled prefix).

## 3. Ticket management (`3-ticket-management.md`)

The core domain model.

- **IDs:** `HS-N`, sequence-backed, never reused even after delete.
- **6 default categories** (customizable): Issue, Bug, Feature, Req Change, Task, Investigation — each with label, shortLabel, color, shortcut key, description.
- **5 priorities:** highest / high / default / low / lowest.
- **7 statuses:** not_started, started, completed, verified, backlog, archive, deleted. Composite filter values: `open`, `non_verified`, `active`.
- **Tags:** bracket syntax `[tag]` in titles auto-extracts. Stored normalized lowercase, displayed Title Case.
- **Notes:** JSON array with auto-generated IDs.
- **Up Next flag** can reopen completed/verified tickets.
- **Batch operations, soft + hard delete, restore.**
- **Auto-cleanup:** configurable (default 3 days for deleted, 30 days for verified/completed).

**Status:** Shipped; conflict resolution UI is cross-cutting with the plugin system (§18).

## 5. Attachments (`5-attachments.md`)

Drag-and-drop files into the detail panel (nested enter/leave counter prevents flicker); files stored as `.hotsheet/attachments/{ticket_number}_{original_name}.{ext}`. Multiple files upload sequentially. Standard MIME types (PNG/JPEG/GIF/SVG/WebP, PDF, text/markdown/JSON, ZIP, HTML/CSS/JS) served correctly; unknown types fall back to octet-stream. Platform-specific "Reveal in Finder" (macOS `open -R`, Windows `explorer /select`, Linux `xdg-open`). Deletion removes DB row + file. Auto-cleanup runs on hard-delete and trash empty.

## 6. Markdown sync & AI-skill generation (`6-markdown-sync.md`)

The bridge from Hot Sheet to AI tools.

- **Exports:** `.hotsheet/worklist.md` (Up Next, 500 ms debounced), `.hotsheet/open-tickets.md` (all open, 5 s debounced). Worklist includes curl examples for status updates and ticket creation.
- **Auto-prioritize** (default on, settings key `auto_order`): when Up Next is empty, the worklist contains instructions for the AI to select the next items itself.
- **Skill generation:** `.claude/skills/hotsheet/SKILL.md` + per-category `hs-{category}/SKILL.md` for Claude Code; `.cursor/rules/hotsheet.mdc`, `.github/prompts/hotsheet.prompt.md`, `.windsurf/rules/hotsheet.md`. Per-category skills parse prefixes like "next" / "up next" / "do next" to set `up_next: true`.
- **Version-gated regeneration:** skills include `<!-- hotsheet-skill-version: N -->`; only regenerated when the code's version bumps. Bump it whenever generated skill content should change.
- **Secret validation:** mutations outside localhost require `X-Hotsheet-Secret` header.
- **Port range pattern:** curl examples in generated skills use 4170–4199 so they survive port fallback.

**Status:** Shipped.

## 7. Backup & restore (`7-backup-restore.md`)

Three-tier automated backups (5-minute × 12 → ~1 hour retention; hourly × 12 → ~12 hours; daily × 7 → ~1 week). Stored as PGLite tar.gz dumps in `.hotsheet/backups/{tier}/` or a custom `backupDir`. "Backup Now" triggers the 5-minute tier and resets its timer. **Preview mode** loads a backup into a temporary PGLite instance with read-only UI (banner + disabled detail panel). Restore creates a safety backup first, closes current DB, loads the backup, and triggers markdown resync. Cleanup combines count-based (maxCount) and time-based (maxAge) pruning. **§7.7 hardening (HS-7888/7889/7891/7894):** every dump issues `CHECKPOINT` first so tarballs round-trip cleanly; startup catches up on overdue tiers (no more skipped daily backups when the user restarts within 24h); open-failure recovery removes a stale `postmaster.pid` and retries before renaming as corrupt, never auto-deletes live data, and logs the underlying PGLite error. **§7.8** documents the `pg_resetwal`-based disaster-recovery runbook for tarballs that pre-date the checkpoint fix. **§7.9 launch-time recovery banner (HS-7899):** when the recovery path falls through to rename+recreate, a `.db-recovery-marker.json` is written in dataDir; the client polls `GET /api/db/recovery-status` on boot and shows a red banner with "Restore from backup…" / "Dismiss" actions until the user responds. Marker survives reload/restart; cleared by `POST /api/db/dismiss-recovery` or by a successful restore.

**Status:** Shipped.

## 9. REST API (`9-api.md`)

Single source of truth for HTTP contracts. Zod validation throughout. All mutations bump a change version consumed by long-poll (`/api/poll`, 30 s timeout, 5 s retry); header `X-Hotsheet-User-Action: true` bumps `last_read_at` (controls the unread dot). - **Dashboard** returns 7/30/90-day KPIs, throughput, CFD, cycle-time scatter.
- **Settings are split** between DB (plugin-only) and file (`/api/file-settings`).

## 10. Desktop app (`10-desktop-app.md`)

Tauri v2 wraps the Node server as a sidecar (see §Tauri below for why). CLI launcher may pre-start the server (`HOTSHEET_SERVER_URL` + `HOTSHEET_SIDECAR_PID` env vars); Tauri alternatively spawns the sidecar itself. Sidecar cleanup: Unix sends SIGTERM then kills the process group; Windows uses `taskkill /T /F`. Welcome screen appears if no prior projects. ## 12. Claude Channel (`12-claude-channel.md`)

The MCP channel gives Claude Code a real-time link to Hot Sheet. Three components: channel server (MCP server, bundled at `dist/channel.js`, listens on local HTTP port, forwards to Claude Code over stdio), Hot Sheet server (detects channel via `.hotsheet/channel-port` file, POSTs events), and Hot Sheet UI (play button in sidebar).

- **Settings:** Experimental tab → Claude Channel toggle. Disabled if Claude CLI < v2.1.80. Stored as `channelEnabled` in `~/.hotsheet/config.json`, with per-project `channel_enabled` fallback.
- **.mcp.json registration:** production uses `node dist/channel.js --data-dir ...`; dev uses `npx tsx src/channel.ts`.
- **Play button:** single-click verifies channel, checks Up Next, flushes pending syncs, sends event. Double-click toggles **auto mode** (immediate trigger, 5 s debounce on Up Next changes, exponential backoff 5→10→20→40→80→120 s on busy, re-trigger on `/channel/done`).
- **Busy/idle status:** spinner + text, 60 s fallback timeout. Heartbeat hooks installed into `~/.claude/settings.json` (PostToolUse=heartbeat, UserPromptSubmit=busy, Stop=idle, 30 s sliding timer).
- **Permission relay:** channel server receives permission_request notifications, exposes via `/permission`; Hot Sheet polls `/api/channel/permission` (long-poll + notify). Overlay shows tool name/description/input preview; buttons Allow / Deny / Dismiss (120 s expiry). Compact popup for inactive project tabs. HS-7266: popup is non-modal — outside clicks are delivered normally to the rest of the UI and an explicit **Minimize** text link (next to the existing **No response needed** link) drops the popup into the owning tab's pulsating blue dot for later attention. The earlier HS-6637 capture-phase outside-click handler was removed because it blocked UI interactions while a permission was pending.
- **Custom commands:** 9-color palette, Lucide icon picker (1693 icons, 24 featured), name, prompt, stored in `custom_commands` settings key. Appear as buttons below play button.

**Status:** Shipped. Cross-refs: §1 (localhost-only), §6 (skill regeneration bump), §15 (custom command targets), §16 (command groups).

## 15. Shell commands (`15-shell-commands.md`)

Custom commands can target **Claude Code** (requires channel) or **Shell** (runs locally). When channel is disabled, new commands default to Shell and are always visible; Claude-target commands hide when channel is disabled. Execution: `POST /api/shell/exec` → `child_process.spawn` with `shell: true`, cwd = project root (parent of `.hotsheet/`). A `shell_command` log entry is created; client shows "Shell running" (same busy indicator as Claude Channel); stdout/stderr captured and appended (separator `---SHELL_OUTPUT---`); client polls `/api/shell/running`. Kill: `POST /api/shell/kill` sends SIGTERM, SIGKILL after 3 s if not exited. `CustomCommand` interface: `name`, `prompt`, `icon`, `color`, `target` (`claude`|`shell`, default `claude`), `autoShowLog` (boolean). Non-zero exit auto-opens the command log regardless of `autoShowLog`.

**Status:** Shipped.

## 18. Plugin system (`18-plugins.md`)

Extensible plugin system for external ticketing backends (GitHub, Linear, Jira, …) and custom UI. Build-time toggle `PLUGINS_ENABLED=false` disables the subsystem. Plugins installed under `~/.hotsheet/plugins/<id>/` (global); bundled plugins copied there if missing or outdated, tracked in `~/.hotsheet/dismissed-plugins.json`. Per-project enable state in DB `plugin_enabled:{id}`; reactivation on config change, sync, action, or push-ticket endpoints.

- **Manifest fields:** id, name, version, description?, author?, entry?, icon? (inline SVG), preferences?, configLayout?.
- **Preference types:** string, boolean, number, select, dropdown, combo. Optional: default, description, required, secret, scope (global|project), options.
- **ConfigLayout items:** preference, divider, spacer, label (id/text/color + `updateConfigLabel`), button (id/label/action/icon?/style?), group (title/collapsed?/items).
- **PluginContext:** config, log, getSetting, setSetting, registerUI, updateConfigLabel.
- **TicketingBackend:** identity, capabilities (create/update/delete/incrementalPull/comments/syncableFields), fieldMappings, CRUD, sync (`pullChanges(since)`, `getRemoteTicket?`), status (`checkConnection`, `getRemoteUrl?`, `shouldAutoSync?`), comments, attachments (`uploadAttachment?`).
- **Sync engine:** push via direct timestamp comparison; pull fetches via `pullChanges`; comments via three-way merge using `note_sync.last_synced_text`; attachments uploaded via backend then posted as markdown link.
- **UI extensions (8 locations):** toolbar, status_bar, sidebar_actions_top, sidebar_actions_bottom, detail_top, detail_bottom, batch_menu, context_menu. **Element types rendered:** button, link. **Declared but not yet rendered:** toggle, switch, segmented_control.
- **Image proxy:** `GET /api/plugins/:id/image-proxy` handles private-repo images with stored PAT.
- **API endpoints:** 27 endpoints covering install, enable/disable/reactivate, sync, push-ticket, action, uninstall, config, schedules.

**Status:** Mostly shipped. **Design only:** `batch_menu` rendering, `toggle`/`switch`/`segmented_control` UI types. Cross-refs: §19 (demo), §20 (secret storage).

## 22. Embedded terminal (`22-terminal.md`)

Adds Terminal tabs to the footer drawer (alongside Commands Log) hosting interactive xterm.js terminals wired to per-project PTYs on the server. The Commands Log tab is icon-only (HS-6474) with a vertical divider before the terminal tabs (HS-6475); Cmd+Shift+Arrow cycles terminal tabs when a terminal is focused (HS-6472, add Alt/Option to route back to project tabs). Each project can run **multiple terminals**: (a) configured default terminals persisted in `settings.terminals` (editable outline list in Settings → Experimental), and (b) dynamic terminals created ad-hoc via a **+** button in the drawer tab strip and closed via an **×** button on the tab. Registry keys sessions by `${secret}::${terminalId}` so multiple terminals coexist. Terminals are server-owned so they survive drawer close / tab switch / browser reload / multi-window attach (tmux-like — multiple windows share the same session per terminal). Spawn is lazy (on first attach); scrollback kept in a 1 MiB (configurable) ring buffer on the server for reattach replay. Default command is a template `command` field per-terminal; `{{claudeCommand}}` resolves to `claude --dangerously-load-development-channels server:hotsheet-channel` when `channelEnabled` is true, else `claude`, else the user's shell. Power toggle (Stop/Start) replaces the earlier Restart button: first click sends SIGHUP (interactive shells ignore SIGTERM but respect SIGHUP, HS-6471), a latched second click prompts before escalating to SIGKILL, Start spawns a fresh PTY. Right-click on a tab opens a context menu with Close Tab / Close Others / Close to the Left / Close to the Right (HS-6470) — configured defaults disable Close Tab and are exempt from the bulk-close actions. The whole feature is **Tauri-only** (HS-6437): non-Tauri clients (plain browser access) hide the drawer tab strip and the Settings → Terminal tab button. Settings live on their own dedicated **Terminal** tab (HS-6337, moved out of Experimental), the legacy `terminal_enabled` checkbox is gone (the feature is simply on in Tauri), and projects default to an empty `terminals` list — there is no implicit default terminal. Transport is WebSocket (`/api/terminal/ws?project=<secret>&terminal=<id>`); native dep is `node-pty` (first non-WASM native addon). Drawer-layout prerequisite (push-up, HS-6262) was delivered separately; HS-6312 added a full-height expand button in the drawer tab strip (state persisted as `drawer_expanded`).

**Out of scope:** per-window private terminals, session recording, SSH / remote targets, split panes. Cross-refs: §4 (drawer layout), §12 (channel gates command substitution), §14 (Commands Log tab sits next to them), §15 (one-shot shell stays separate).

**Status:** Shipped — full v1 end-to-end plus HS-7827 (drawer terminal tabs are drag-to-reorderable: each `.drawer-terminal-tab` is `draggable="true"` with `dragstart` / `dragover` / `drop` handlers wired by `attachTabDragHandlers(tabBtn, terminalId)`; module-level `tabDragFromId` tracks drag state; on drop, `reorderTabAfterDrop` re-appends tabs + matching panes in the new order and PATCHes `/file-settings` with the configured-id-only subset of the new strip order. ## 37. Quit confirmation when terminals are running (`37-quit-confirm.md`)

HS-7591 (req-change spec) + HS-7596 (implementation). macOS Terminal.app-style "Ask before closing" gating across every quit path so users don't silently lose running work. Three modes per project (`confirm_quit_with_running_terminals: 'always' | 'never' | 'with-non-exempt-processes'`, default `'with-non-exempt-processes'`) + a user-editable exempt list (`quit_confirm_exempt_processes`, default `{screen, tmux, less, more, view, mandoc, tail, log, top, htop}`). One-level-deeper rule for shell-rooted PTYs: when the configured `command` IS a shell (`bash`, `zsh`, `fish`, `sh`, `dash`, `ash`, `ksh`, `tcsh`, `pwsh`, `powershell`, `cmd`, `cmd.exe`), look at the foreground child of the shell to decide whether to prompt; the shell itself is treated as exempt (idle login shell). When the configured command is NOT a shell (e.g. `claude`, `npm run dev`), the root process is what's evaluated.

**Detection** (server-side, Unix only for v1 — Windows safe-defaults to "prompt"). `inspectForegroundProcess(rootPid, exemptList)` runs `ps -o pid,ppid,comm -A` once, parses to a parent→children map, walks the deepest descendant chain (highest-pid child at each level), then `pickForegroundProcess` applies the §37.3 one-level-deeper rule + exempt-list lookup (case-insensitive basename match). On any lookup failure (process exited mid-check, ps unavailable) returns the safe-default-prompt info — users prefer one extra confirmation over a silent kill. Pure helpers in `src/terminals/processInspect.ts` (parsePsOutput / descendantChain / pickForegroundProcess) covered by 22 unit tests.

**Routes.** `GET /api/terminal/foreground-process?terminalId=` returns the per-terminal info. `GET /api/projects/quit-summary` aggregates across every registered project: walks every alive PTY, inspects its foreground process, returns `{projects: [{secret, name, confirmMode, entries: [{terminalId, label, foregroundCommand, isShell, isExempt}]}]}`. The route does NOT decide whether to prompt — the client owns that policy because the prompt context (Tauri close vs. CLI --close vs. stale-instance cleanup) varies.

**Decision** (client-side, pure). `evaluateQuitDecision(summary)` in `src/client/quitConfirm.tsx` applies §37.5 across all projects: prompt fires if any project is `'always'` OR if any project's `'with-non-exempt-processes'` resolves to "has at least one non-exempt entry"; `'never'` projects don't trigger but contribute their alive entries to the displayed list (so users see what they're killing across every project). 9 unit tests in `quitConfirm.test.ts`.

**Tauri quit-handler.** `tauri::WindowEvent::CloseRequested` in `src-tauri/src/lib.rs` checks a `QuitConfirmed: AtomicBool` state. First close attempt: `api.prevent_close()` + emit `quit-confirm-requested` event. The JS frontend (`initQuitConfirm` listener via `__TAURI__.event.listen`) runs the confirm flow + on "Quit Anyway" invokes the new `confirm_quit` Tauri command. `confirm_quit` sets the AtomicBool to true and re-issues `window.close()`; the second CloseRequested fire sees the flag and lets the close proceed. `RunEvent::Exit` then kills the sidecar via the existing SIGTERM path.

**CLI `--close`.** Extends `handleClose` in `src/cli.ts` with a `confirmCloseAgainstQuitSummary` step before the DELETE: fetches /api/projects/quit-summary, filters to the project being closed, applies the §37.5 logic, prompts via stdin (`y/N`). New `--force` flag skips the prompt for non-interactive use.

**Settings UI.** Settings → Terminal gains a "Quit confirmation" section (`src/client/quitConfirmSettingsUI.tsx`) with the three radio modes + a textarea for the exempt list (one basename per line) + a "Reset exempt list to defaults" link. Saves are debounced + write through PATCH /api/file-settings. The exempt list is stored as JSON array (added to `JSON_VALUE_KEYS` in file-settings.ts).

**Don't ask again.** The dialog has a "Don't ask again for any project" checkbox that, on Quit Anyway, flips every project's `confirm_quit_with_running_terminals` to `'never'`. Power-user opt-out without leaving the dialog.

**Out of scope (v1):** per-terminal kill confirm (handled by existing tab-close/Stop flows), session save+restore on relaunch, graceful SIGINT/SIGTERM-then-SIGHUP escalation in the dialog (existing teardown signal flow handles it), cross-window aggregation across separate Tauri processes. Stale-instance cleanup intentionally bypasses since that path is programmatic + the user is already quitting through a new window.

**HS-7789 follow-up.** Quit-confirm dialog showed the raw `dyn-…` id for dynamic terminals. Root cause: `/api/projects/quit-summary` resolved labels via `listTerminalConfigs(dataDir)` only — that returns persisted (settings-backed) terminals, and dynamic terminals created via `POST /api/terminal/create` keep their friendly name in the in-memory `dynamicConfigs` map in `src/routes/terminal.ts`. ## 41. Backup JSON co-save (`41-backup-json-cosave.md`)

HS-7893. Every scheduled backup writes a versioned, gzipped JSON snapshot of every row in every Hot Sheet table next to the existing PGLite tarball: `backup-<TS>.json.gz` alongside `backup-<TS>.tar.gz`. Pure escape hatch — when a tarball PANICs on `loadDataDir` (the HS-7891 incident pattern) the JSON gives a developer or future-Claude a path to rebuild the schema and re-insert rows. **Not** wired into the Settings → Backups restore UI; restore continues to use the tarball.

**Format.** `{ schemaVersion: 1, exportedAt: ISO, tables: { tickets: [...], attachments: [...], settings, stats_snapshots, command_log, ticket_sync, sync_outbox, note_sync, feedback_drafts } }`. `SCHEMA_VERSION` is a hardcoded constant in `src/db/connection.ts`, bumped manually when `initSchema` adds/removes/renames a column or table — readers compare it against the current code's value to decide whether the rows still load as-is. Hard-coded TABLES list in `src/dbJsonExport.ts` mirrors `initSchema` (deterministic across PGLite versions, no `information_schema` system rows). A missing table yields `[]` rather than throwing so the rescue path stays available mid-migration.

**Atomic write.** `writeJsonExportAtomically(path, exportData)` does `gzipSync` → `openSync('<path>.tmp')` → `writeSync` → `fsyncSync` → `closeSync` → `renameSync('<path>.tmp', '<path>')`. A crash mid-write leaves either the previous file or nothing — never a partial. `.tmp` orphans are unlinked on rename failure.

**Pruning.** `pruneBackups()` deletes the `.json.gz` sibling whenever it deletes the `.tar.gz` so the two stay in lockstep across the same retention/maxAge policy.

**Attachment blobs out of scope.** Attachment row metadata is exported, but the binary files under `.hotsheet/attachments/<sha>` are still not copied anywhere (true for the tarball too). Tracked in HS-7900 for a separate attachment-backup investigation.

**Files.** `src/dbJsonExport.ts` (new — `buildJsonExport`, `writeJsonExportAtomically`, `jsonSiblingFilename`), `src/db/connection.ts` (`SCHEMA_VERSION`), `src/backup.ts` (call site + prune sibling). Tests: `src/dbJsonExport.test.ts` + integration assertion in `src/backup.test.ts`.

**Status:** Shipped. Cross-refs: §7.7 (CHECKPOINT-before-dump + open-failure recovery hardening from the same incident), §7.8 (disaster-recovery runbook the JSON gives an alternative path into).

## 43. Attachment backups (`43-attachment-backups.md`)

HS-7900 design / HS-7929 implementation. Closes the gap left by §7 (PGLite tarball) and §41 (JSON co-save), neither of which carries the binary blobs under `.hotsheet/attachments/`. Hash-addressed centralised store: `<backupRoot>/attachments/<sha256-hex>` holds each blob exactly once; per-backup `backup-<TS>.attachments.json` manifest sibling lists the `{attachmentId, ticketId, originalName, storedName, sha, size}` entries the backup needs. Daily orphan GC walks every manifest's union of shas and deletes any blob not referenced. Manifest re-analysis rebuilds a missing manifest from the JSON co-save sibling at startup (skipping tarballs younger than 24h to avoid hashing live attachments on every boot). Restore extends the existing `POST /api/backups/restore` flow: copy each manifest entry's hash blob into `<dataDir>/attachments/<storedName>`, update the restored row's `stored_path`, and append `-restored-<TS>` if a live file with the same storedName has different content.

**Decision rationale.** Of the four options HS-7900 enumerated, hash-addressed dedup (option 3) was chosen per the user's direction in the ticket notes. Option 1 (mirror to `.hotsheet/attachments-backup/`) lives outside the backup root so a custom `backupDir` (e.g. Google Drive per HS-7891) wouldn't cover it. Option 2 (bundle into the tarball) re-bundles every blob every 5 min — death by a thousand cuts on slow remote drives. Option 4 (hard-link tree) breaks across filesystems. Single shared `<backupRoot>/attachments/` (not per-tier) maximises dedup and gives the GC one location to walk.

**Status:** Mostly shipped (HS-7929). Backup-side capture, daily GC, restore re-hydration are all live; only manifest re-analysis at startup (§43.7) is deferred to **HS-7937**.

**Shipped (HS-7929).** New `src/attachmentBackup.ts` (300+ lines) with: `manifestSiblingFilename` for the `<base>.attachments.json` naming convention, streaming `hashFile` via `pipeline(createReadStream, hash)` so 200 MB blobs don't blow up the heap, `ensureBlobInStore` (link-then-copy fallback for cross-fs `backupDir` like Google Drive, atomic via tmp + rename), `writeManifestAtomically` mirroring HS-7893's pattern, `buildAttachmentManifest` querying the live `attachments` table + dispatching one entry per row, `runAttachmentGc` walking every manifest's `entries[].sha` union and reclaiming orphans (aborts on any parse failure to avoid orphaning live data), `restoreAttachmentsFromManifest` copying hash blobs back into `<dataDir>/attachments/<storedName>` with `-restored-<TS>` suffix on content collision and idempotent skip on hash match. `src/backup.ts` `createBackup` calls `buildAttachmentManifest` after the JSON co-save block; `pruneBackups` calls `deleteManifestSibling` to keep manifests in lockstep with tarballs; `initBackupScheduler` schedules a 24h GC interval (with a 30s startup delay so the initial backup catch-up settles first); `restoreBackup` reads the manifest sibling on restore and updates each restored row's `stored_path` to wherever the blob landed (handling the `-restored-<TS>` suffix case). Tests: `src/attachmentBackup.test.ts` (23 cases — naming, hashing, blob store, manifest IO, build, GC, restore name-collision behaviour) + 1 end-to-end integration in `src/backup.test.ts`.

## 44. WASM `pg_resetwal` design spike (`44-wasm-pg-resetwal.md`)

HS-7901. Investigates whether Hot Sheet can ship its own WASM-compiled `pg_resetwal` so the §42 repair flow no longer depends on a system Postgres install. Surveys four options: (1) extend PGLite's bundle to also export `pg_resetwal_main`, (2) standalone lazy-loaded `pg-resetwal.wasm` built from the postgres source, (3) re-implement the relevant `pg_control` rewrite + WAL zero in TypeScript, (4) status quo (system binary).

**Verdict.** Defer. §42's system-binary path is good enough today and no in-the-wild incident has surfaced from a user who couldn't install `postgresql@17`. Re-evaluate when (a) PGLite upstream ships utility-binary support — collapsing option 1 to a `npm update` — or (b) a real user hits the no-admin / Tauri-only wall. If we have to act sooner, Option 2 (standalone lazy-loaded WASM) is the recommended path: beats option 1 on time-to-value (no upstream gatekeeper), beats option 3 on safety (uses the actual postgres source, not a hand-port).

**Key trade-offs.** Option 1 auto-tracks PGLite's PostgreSQL major version but requires forking + maintaining a patched build OR upstreaming. Option 2 has the highest implementation cost (3–5 weeks) but stays independent of PGLite's release cadence. Option 3 has the lowest binary-size hit but is format-locked to a specific PG major version and carries only the happy path of a tool that has accumulated decades of edge-case fixes. Filesystem shim is the same regardless: re-use PGLite's `loadDataDir` / `dumpDataDir` to bridge memfs ↔ host-fs, call `pg_resetwal_main(["pg_resetwal", "-f", "/data"])`, write modified files back.

**Status:** Design only. Recommendation = defer.

## 45. PGLite robustness — cleaner shutdown (`45-pglite-robustness.md`)

HS-7902. Audit + design for closing the gap that drove the HS-7888..7894 incident chain at its source instead of just papering over it. Today every shutdown path (`/api/shutdown`, SIGINT, SIGTERM, Tauri close) ends in `process.exit()` without ever calling `db.close()`, so PGLite's `postmaster.pid` is always stale on next launch and HS-7888's reactive mitigation has to clean up. Design proposes a shared `gracefulShutdown(reason)` helper in `src/lifecycle.ts` that (1) closes the HTTP server so a CHECKPOINT doesn't race in-flight writes, (2) destroys PTYs (already shipped), (3) `await db.close()` per cached PGLite instance — PGLite's own close path CHECKPOINTs + removes the pid, and (4) removes the instance lockfile. Wired into `/api/shutdown` (replaces the existing 500ms `setTimeout` race), `process.on('SIGINT'|'SIGTERM')` (returns a Promise; second signal escalates to `exit(1)` so a hung close can't trap the user), and Tauri close (already routes through `/api/shutdown`).

**Live-cluster CHECKPOINT cadence.** Decision = no extra timer. The 5-minute backup tier already runs `CHECKPOINT` before each `dumpDataDir` (HS-7891), so the worst-case crash data-loss window is the same as the user's existing backup exposure. Adding a separate 60s timer would write to disk on idle machines for no benefit.

**Out-of-scope spikes** filed alongside the implementation: HS-7932 verifies PGLite's `fsync` round-trips through to host fs on each platform (concern: WASM ↔ host-fs bridge could silently no-op `fsyncSync`); HS-7933 benchmarks tighter `checkpoint_timeout = 60s` + `max_wal_size = 256 MB` to see if a smaller crash-recovery window is worth the disk-write delta.

**Status:** Partial — graceful close shipped (HS-7931). fsync verification spike (HS-7932) and checkpoint-timeout benchmark (HS-7933) still open.

**Shipped (HS-7931).** New `src/lifecycle.ts` exports `gracefulShutdown(reason)` + `registerHttpServerForShutdown(server)`. Pipeline: close HTTP → destroy PTYs → `closeAllDatabases()` (new export in `src/db/connection.ts`) → remove instance lockfile. Idempotent (concurrent callers share the same promise). Wired into `/api/shutdown` (replaces the previous 500ms `setTimeout(process.exit)` race) and the SIGINT/SIGTERM handlers in `cli.ts` (a second identical signal forces `exit(1)` so a hung close can't trap the user). Synchronous `process.on('exit')` stays as the safety-net lockfile remover. Tests: `src/lifecycle.test.ts` (8 cases — order, idempotence, per-step error tolerance) + `src/db/connection.test.ts` "closeAllDatabases" block (cache-clear contract + close-failure tolerance).

**Cross-refs:** §7.7 (backup hardening from same incident chain), §7.9 (HS-7899 launch banner — kicks in if graceful close still ends up in the rename path), §41 (JSON co-save), §42 (Database Repair — manual recovery for cases where prevention here doesn't help), §44 (WASM `pg_resetwal` — automation of §42 if ever shipped).

## 21. Feedback notes (`21-feedback.md`)

AI tools request user feedback by adding a note to a ticket whose text begins with a recognized prefix (checked only on the most recent note):

- `FEEDBACK NEEDED:` — shows dialog when ticket is opened.
- `IMMEDIATE FEEDBACK NEEDED:` — auto-selects the ticket in the project's tab and opens the dialog.

Dialog shows prompt as markdown, one or more textareas, file attachments, and three buttons: **Later** (muted link, dismiss), **No Response Needed** (adds `NO RESPONSE NEEDED` note, clears state), **Submit** (creates note with response, uploads attachments, notifies channel). Click-outside dismisses like Later. Dialog auto-shows **once per detail-panel open** (tracked by note ID). "Provide Feedback" link button appears below a feedback-prefix note if it's the most recent.

**HS-6998 click-to-insert inline responses** (§21.2.1): earlier auto-split-by-list heuristic (first qualifying list → per-item textareas) misread prompts where the list was an A/B/C/D options menu rather than independent questions. Current approach does NOT guess: the prompt is split into top-level markdown blocks (paragraphs / lists / headings / code) via `parseFeedbackBlocks` in `src/client/feedbackParser.ts`, each block renders as an accent-bordered pill, and between every pair of blocks (and after the last) a muted `+ Add response here` button lets the user insert an inline textarea at that position. Inline responses have a `×` remove button; multiple responses per slot are allowed. A catch-all textarea always sits at the bottom. Submit semantics: (a) no inline responses used → catch-all text is the note body verbatim (no quoting); (b) any inline responses used → `combineQuotedResponse(blocks, inlineResponses, catchAll)` re-emits the whole prompt as markdown blockquotes (`> ` prefix on every line of every block) with the user's inline responses interleaved un-quoted in their slots and the catch-all appended un-quoted at the end. Empty-whitespace inline responses are dropped rather than rendered as placeholders. Covered by unit tests in `feedbackParser.test.ts`.

**HS-7558 per-list-item insertion points** (§21.2.2): the v1 always rendered a list as a single block, so a numbered list of questions only got ONE `+ Add response here` slot at the end of the whole list — users wanted to answer question-by-question. New `shouldSplitListIntoItems(list, isFirstBlock)` heuristic in `parseFeedbackBlocks` splits a top-level list into one block per item when EITHER (a) the list is the first meaningful block of the prompt (the screenshot scenario — leading numbered question lists are almost always question sets regardless of punctuation), OR (b) every item's first line ends with `.`, `?`, or `!`. Sub-bullets nested under a list item stay grouped with their parent so a "1. Top question? — sub a — sub b" block keeps the parent + clarifications together. `combineQuotedResponse` is unchanged — `blockIndex` indexes into whatever block list the parser returns, so per-item blocks slot in transparently. 4 new unit tests in `feedbackParser.test.ts` cover the four cases (leading list always splits, mid-prompt sentence-punctuated list splits, mid-prompt option list stays grouped, partial-punctuation list stays grouped) plus a sub-bullet-preservation test.

**HS-7599 Save Draft + don't-close-on-clickaway** (§21.2.3): the dialog now persists in-progress responses to a new `feedback_drafts` table when the user clicks the new Save Draft button, and click-outside-the-dialog dismissal is gated on whether ANY input has text (catch-all OR any inline textarea). ## Tauri architecture & setup (`tauri-architecture.md`, `tauri-setup.md`)

**Both missing from CLAUDE.md reading order — add them.**

**Architecture:** Tauri v2 wraps Node as a sidecar because PGLite (WASM) needs filesystem access — single-binary compilers break it. Layout: `src-tauri/{src, tauri.conf.json, Cargo.toml, Entitlements.plist, capabilities, loading, resources, binaries, server, icons}`. Three launch flows: (1) double-click (no `--data-dir`) → welcome.html → CLI install wizard; (2) CLI launch on macOS (complex due to JIT/filesystem restrictions) → CLI script resolves app name, creates a stub `.app` in `.hotsheet/`, starts Node in background, writes URL+PID to `/tmp/hotsheet-server-{hash}.info`, `open -a` with stub app → stub launcher reads `/tmp`, sets `HOTSHEET_SERVER_URL`+`HOTSHEET_SIDECAR_PID`, execs real Tauri binary with `--data-dir`; (3) Tauri with `--data-dir` (fallback/dev) → spawns sidecar, waits for "running at" on stdout. Sidecar lifecycle tracked via `SidecarPid` managed state, killed on `RunEvent::Exit` (Unix: `libc::kill(-pid, SIGTERM)`; Windows: `taskkill /T /F`). Build pipeline: `scripts/build-sidecar.sh` downloads Node v20, runs `npm run build`, copies `dist/` + assets into `src-tauri/server/`; `tauri build` produces `.app`/`.AppImage`/`.msi`. CI/CD (`release-desktop.yml`) builds 4 targets, signs macOS, signs updates, creates draft GitHub Release. macOS entitlements: `allow-jit`, `allow-unsigned-executable-memory`, `disable-library-validation` (required for signed V8/WASM).
