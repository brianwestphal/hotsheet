## 1. Product vision & foundational decisions

Hot Sheet is a local-first, developer-focused ticket tracker that lives next to a codebase. A CLI (`hotsheet`) spins up an embedded HTTP server + browser UI; data lives in `.hotsheet/` beside the project; a Tauri wrapper offers a native desktop experience. The product's central bet is **AI-integration**: tickets are auto-exported to markdown worklists, and an MCP channel lets Claude Code work through the worklist turn-by-turn.

- **Non-functional requirements** (¬ß1): strict data locality, debounces tuned for responsiveness (worklist 500 ms, open-tickets 5 s), parameterized SQL only, soft-delete everywhere, 3-tier auto backups, signed desktop updates, cross-platform (macOS, Linux, Windows).
- **Build outputs** (¬ß1): `dist/cli.js`, `dist/channel.js`, `dist/client/app.global.js`, `dist/client/styles.css`.
- **Status:** Shipped.

## 2. Data & storage (`2-data-storage.md`)

PGLite embedded Postgres with automatic additive migrations, tables for tickets (HS-prefixed auto-increment IDs, never reused), attachments, stats snapshots, command log, settings (plugin-only now ‚Äî project config moved to `.hotsheet/settings.json`), plus sync tables (`ticket_sync`, `sync_outbox`, `note_sync`). Notes stored as JSON array `{ id, text, created_at }`. File-based settings.json carries `appName`, `backupDir`, `appIcon`, `ticketPrefix`, `secret`, `port`, UI layout keys. A `.hotsheet/hotsheet.lock` file (pid/startedAt) enforces single-instance per data dir; `~/.hotsheet/instance.json` + `projects.json` support multi-project.

**Status:** Shipped (full schema, migrations, stale-lock cleanup, multi-project registry). Cross-refs: ¬ß3 (notes), ¬ß7 (backupDir), ¬ß18 (plugin_enabled prefix).

## 3. Ticket management (`3-ticket-management.md`)

The core domain model.

- **IDs:** `HS-N`, sequence-backed, never reused even after delete.
- **6 default categories** (customizable): Issue, Bug, Feature, Req Change, Task, Investigation ‚Äî each with label, shortLabel, color, shortcut key, description.
- **5 priorities:** highest / high / default / low / lowest.
- **7 statuses:** not_started, started, completed, verified, backlog, archive, deleted. Composite filter values: `open`, `non_verified`, `active`.
- **Tags:** bracket syntax `[tag]` in titles auto-extracts. Stored normalized lowercase, displayed Title Case.
- **Notes:** JSON array with auto-generated IDs.
- **Up Next flag** can reopen completed/verified tickets.
- **Batch operations, soft + hard delete, restore.**
- **Auto-cleanup:** configurable (default 3 days for deleted, 30 days for verified/completed).

**Status:** Shipped; conflict resolution UI is cross-cutting with the plugin system (¬ß18).

## 7. Backup & restore (`7-backup-restore.md`)

Three-tier automated backups (5-minute √ó 12 ‚Üí ~1 hour retention; hourly √ó 12 ‚Üí ~12 hours; daily √ó 7 ‚Üí ~1 week). Stored as PGLite tar.gz dumps in `.hotsheet/backups/{tier}/` or a custom `backupDir`. "Backup Now" triggers the 5-minute tier and resets its timer. **Preview mode** loads a backup into a temporary PGLite instance with read-only UI (banner + disabled detail panel). Restore creates a safety backup first, closes current DB, loads the backup, and triggers markdown resync. Cleanup combines count-based (maxCount) and time-based (maxAge) pruning. **¬ß7.7 hardening (HS-7888/7889/7891/7894):** every dump issues `CHECKPOINT` first so tarballs round-trip cleanly; startup catches up on overdue tiers (no more skipped daily backups when the user restarts within 24h); open-failure recovery removes a stale `postmaster.pid` and retries before renaming as corrupt, never auto-deletes live data, and logs the underlying PGLite error. **¬ß7.8** documents the `pg_resetwal`-based disaster-recovery runbook for tarballs that pre-date the checkpoint fix. **¬ß7.9 launch-time recovery banner (HS-7899):** when the recovery path falls through to rename+recreate, a `.db-recovery-marker.json` is written in dataDir; the client polls `GET /api/db/recovery-status` on boot and shows a red banner with "Restore from backup‚Ä¶" / "Dismiss" actions until the user responds. Marker survives reload/restart; cleared by `POST /api/db/dismiss-recovery` or by a successful restore.

**Status:** Shipped.

## 12. Claude Channel (`12-claude-channel.md`)

The MCP channel gives Claude Code a real-time link to Hot Sheet. Three components: channel server (MCP server, bundled at `dist/channel.js`, listens on local HTTP port, forwards to Claude Code over stdio), Hot Sheet server (detects channel via `.hotsheet/channel-port` file, POSTs events), and Hot Sheet UI (play button in sidebar).

- **Settings:** Experimental tab ‚Üí Claude Channel toggle. Disabled if Claude CLI < v2.1.80. Stored as `channelEnabled` in `~/.hotsheet/config.json`, with per-project `channel_enabled` fallback.
- **.mcp.json registration:** production uses `node dist/channel.js --data-dir ...`; dev uses `npx tsx src/channel.ts`.
- **Play button:** single-click verifies channel, checks Up Next, flushes pending syncs, sends event. Double-click toggles **auto mode** (immediate trigger, 5 s debounce on Up Next changes, exponential backoff 5‚Üí10‚Üí20‚Üí40‚Üí80‚Üí120 s on busy, re-trigger on `/channel/done`).
- **Busy/idle status:** spinner + text, 60 s fallback timeout. Heartbeat hooks installed into `~/.claude/settings.json` (PostToolUse=heartbeat, UserPromptSubmit=busy, Stop=idle, 30 s sliding timer).
- **Permission relay:** channel server receives permission_request notifications, exposes via `/permission`; Hot Sheet polls `/api/channel/permission` (long-poll + notify). Overlay shows tool name/description/input preview; buttons Allow / Deny / Dismiss (120 s expiry). Compact popup for inactive project tabs. HS-7266: popup is non-modal ‚Äî outside clicks are delivered normally to the rest of the UI and an explicit **Minimize** text link (next to the existing **No response needed** link) drops the popup into the owning tab's pulsating blue dot for later attention. The earlier HS-6637 capture-phase outside-click handler was removed because it blocked UI interactions while a permission was pending.
- **Custom commands:** 9-color palette, Lucide icon picker (1693 icons, 24 featured), name, prompt, stored in `custom_commands` settings key. Appear as buttons below play button.

**Status:** Shipped. Cross-refs: ¬ß1 (localhost-only), ¬ß6 (skill regeneration bump), ¬ß15 (custom command targets), ¬ß16 (command groups).

## 22. Embedded terminal (`22-terminal.md`)

Adds Terminal tabs to the footer drawer (alongside Commands Log) hosting interactive xterm.js terminals wired to per-project PTYs on the server. The Commands Log tab is icon-only (HS-6474) with a vertical divider before the terminal tabs (HS-6475); Cmd+Shift+Arrow cycles terminal tabs when a terminal is focused (HS-6472, add Alt/Option to route back to project tabs). Each project can run **multiple terminals**: (a) configured default terminals persisted in `settings.terminals` (editable outline list in Settings ‚Üí Experimental), and (b) dynamic terminals created ad-hoc via a **+** button in the drawer tab strip and closed via an **√ó** button on the tab. Registry keys sessions by `${secret}::${terminalId}` so multiple terminals coexist. Terminals are server-owned so they survive drawer close / tab switch / browser reload / multi-window attach (tmux-like ‚Äî multiple windows share the same session per terminal). ## 37. Quit confirmation when terminals are running (`37-quit-confirm.md`)

HS-7591 (req-change spec) + HS-7596 (implementation). macOS Terminal.app-style "Ask before closing" gating across every quit path so users don't silently lose running work. Three modes per project (`confirm_quit_with_running_terminals: 'always' | 'never' | 'with-non-exempt-processes'`, default `'with-non-exempt-processes'`) + a user-editable exempt list (`quit_confirm_exempt_processes`, default `{screen, tmux, less, more, view, mandoc, tail, log, top, htop}`). One-level-deeper rule for shell-rooted PTYs: when the configured `command` IS a shell (`bash`, `zsh`, `fish`, `sh`, `dash`, `ash`, `ksh`, `tcsh`, `pwsh`, `powershell`, `cmd`, `cmd.exe`), look at the foreground child of the shell to decide whether to prompt; the shell itself is treated as exempt (idle login shell). When the configured command is NOT a shell (e.g. `claude`, `npm run dev`), the root process is what's evaluated.

**Detection** (server-side, Unix only for v1 ‚Äî Windows safe-defaults to "prompt"). `inspectForegroundProcess(rootPid, exemptList)` runs `ps -o pid,ppid,comm -A` once, parses to a parent‚Üíchildren map, walks the deepest descendant chain (highest-pid child at each level), then `pickForegroundProcess` applies the ¬ß37.3 one-level-deeper rule + exempt-list lookup (case-insensitive basename match). On any lookup failure (process exited mid-check, ps unavailable) returns the safe-default-prompt info ‚Äî users prefer one extra confirmation over a silent kill. Pure helpers in `src/terminals/processInspect.ts` (parsePsOutput / descendantChain / pickForegroundProcess) covered by 22 unit tests.

**Routes.** `GET /api/terminal/foreground-process?terminalId=` returns the per-terminal info. `GET /api/projects/quit-summary` aggregates across every registered project: walks every alive PTY, inspects its foreground process, returns `{projects: [{secret, name, confirmMode, entries: [{terminalId, label, foregroundCommand, isShell, isExempt}]}]}`. The route does NOT decide whether to prompt ‚Äî the client owns that policy because the prompt context (Tauri close vs. CLI --close vs. stale-instance cleanup) varies.

**Decision** (client-side, pure). `evaluateQuitDecision(summary)` in `src/client/quitConfirm.tsx` applies ¬ß37.5 across all projects: prompt fires if any project is `'always'` OR if any project's `'with-non-exempt-processes'` resolves to "has at least one non-exempt entry"; `'never'` projects don't trigger but contribute their alive entries to the displayed list (so users see what they're killing across every project). 9 unit tests in `quitConfirm.test.ts`.

## 41. Backup JSON co-save (`41-backup-json-cosave.md`)

HS-7893. Every scheduled backup writes a versioned, gzipped JSON snapshot of every row in every Hot Sheet table next to the existing PGLite tarball: `backup-<TS>.json.gz` alongside `backup-<TS>.tar.gz`. Pure escape hatch ‚Äî when a tarball PANICs on `loadDataDir` (the HS-7891 incident pattern) the JSON gives a developer or future-Claude a path to rebuild the schema and re-insert rows. **Not** wired into the Settings ‚Üí Backups restore UI; restore continues to use the tarball.

**Format.** `{ schemaVersion: 1, exportedAt: ISO, tables: { tickets: [...], attachments: [...], settings, stats_snapshots, command_log, ticket_sync, sync_outbox, note_sync, feedback_drafts } }`. `SCHEMA_VERSION` is a hardcoded constant in `src/db/connection.ts`, bumped manually when `initSchema` adds/removes/renames a column or table ‚Äî readers compare it against the current code's value to decide whether the rows still load as-is. Hard-coded TABLES list in `src/dbJsonExport.ts` mirrors `initSchema` (deterministic across PGLite versions, no `information_schema` system rows). A missing table yields `[]` rather than throwing so the rescue path stays available mid-migration.

**Atomic write.** `writeJsonExportAtomically(path, exportData)` does `gzipSync` ‚Üí `openSync('<path>.tmp')` ‚Üí `writeSync` ‚Üí `fsyncSync` ‚Üí `closeSync` ‚Üí `renameSync('<path>.tmp', '<path>')`. A crash mid-write leaves either the previous file or nothing ‚Äî never a partial. `.tmp` orphans are unlinked on rename failure.

**Pruning.** `pruneBackups()` deletes the `.json.gz` sibling whenever it deletes the `.tar.gz` so the two stay in lockstep across the same retention/maxAge policy.

**Attachment blobs out of scope.** Attachment row metadata is exported, but the binary files under `.hotsheet/attachments/<sha>` are still not copied anywhere (true for the tarball too). Tracked in HS-7900 for a separate attachment-backup investigation.

**Files.** `src/dbJsonExport.ts` (new ‚Äî `buildJsonExport`, `writeJsonExportAtomically`, `jsonSiblingFilename`), `src/db/connection.ts` (`SCHEMA_VERSION`), `src/backup.ts` (call site + prune sibling). Tests: `src/dbJsonExport.test.ts` + integration assertion in `src/backup.test.ts`.

**Status:** Shipped. Cross-refs: ¬ß7.7 (CHECKPOINT-before-dump + open-failure recovery hardening from the same incident), ¬ß7.8 (disaster-recovery runbook the JSON gives an alternative path into).

## 43. Attachment backups (`43-attachment-backups.md`)

HS-7900 design / HS-7929 implementation. Closes the gap left by ¬ß7 (PGLite tarball) and ¬ß41 (JSON co-save), neither of which carries the binary blobs under `.hotsheet/attachments/`. Hash-addressed centralised store: `<backupRoot>/attachments/<sha256-hex>` holds each blob exactly once; per-backup `backup-<TS>.attachments.json` manifest sibling lists the `{attachmentId, ticketId, originalName, storedName, sha, size}` entries the backup needs. Daily orphan GC walks every manifest's union of shas and deletes any blob not referenced. Manifest re-analysis rebuilds a missing manifest from the JSON co-save sibling at startup (skipping tarballs younger than 24h to avoid hashing live attachments on every boot). Restore extends the existing `POST /api/backups/restore` flow: copy each manifest entry's hash blob into `<dataDir>/attachments/<storedName>`, update the restored row's `stored_path`, and append `-restored-<TS>` if a live file with the same storedName has different content.

**Decision rationale.** Of the four options HS-7900 enumerated, hash-addressed dedup (option 3) was chosen per the user's direction in the ticket notes. Option 1 (mirror to `.hotsheet/attachments-backup/`) lives outside the backup root so a custom `backupDir` (e.g. Google Drive per HS-7891) wouldn't cover it. Option 2 (bundle into the tarball) re-bundles every blob every 5 min ‚Äî death by a thousand cuts on slow remote drives. Option 4 (hard-link tree) breaks across filesystems. Single shared `<backupRoot>/attachments/` (not per-tier) maximises dedup and gives the GC one location to walk.

**Status:** Shipped. HS-7929 + HS-7937 together cover backup-side capture, daily GC, restore re-hydration, AND startup manifest re-analysis.

**Shipped (HS-7929).** New `src/attachmentBackup.ts` (300+ lines) with: `manifestSiblingFilename` for the `<base>.attachments.json` naming convention, streaming `hashFile` via `pipeline(createReadStream, hash)` so 200 MB blobs don't blow up the heap, `ensureBlobInStore` (link-then-copy fallback for cross-fs `backupDir` like Google Drive, atomic via tmp + rename), `writeManifestAtomically` mirroring HS-7893's pattern, `buildAttachmentManifest` querying the live `attachments` table + dispatching one entry per row, `runAttachmentGc` walking every manifest's `entries[].sha` union and reclaiming orphans (aborts on any parse failure to avoid orphaning live data), `restoreAttachmentsFromManifest` copying hash blobs back into `<dataDir>/attachments/<storedName>` with `-restored-<TS>` suffix on content collision and idempotent skip on hash match. `src/backup.ts` `createBackup` calls `buildAttachmentManifest` after the JSON co-save block; `pruneBackups` calls `deleteManifestSibling` to keep manifests in lockstep with tarballs; `initBackupScheduler` schedules a 24h GC interval (with a 30s startup delay so the initial backup catch-up settles first); `restoreBackup` reads the manifest sibling on restore and updates each restored row's `stored_path` to wherever the blob landed (handling the `-restored-<TS>` suffix case). ## 44. WASM `pg_resetwal` design spike (`44-wasm-pg-resetwal.md`)

HS-7901. Investigates whether Hot Sheet can ship its own WASM-compiled `pg_resetwal` so the ¬ß42 repair flow no longer depends on a system Postgres install. ## 45. PGLite robustness ‚Äî cleaner shutdown (`45-pglite-robustness.md`)

HS-7902. Audit + design for closing the gap that drove the HS-7888..7894 incident chain at its source instead of just papering over it. Today every shutdown path (`/api/shutdown`, SIGINT, SIGTERM, Tauri close) ends in `process.exit()` without ever calling `db.close()`, so PGLite's `postmaster.pid` is always stale on next launch and HS-7888's reactive mitigation has to clean up. Design proposes a shared `gracefulShutdown(reason)` helper in `src/lifecycle.ts` that (1) closes the HTTP server so a CHECKPOINT doesn't race in-flight writes, (2) destroys PTYs (already shipped), (3) `await db.close()` per cached PGLite instance ‚Äî PGLite's own close path CHECKPOINTs + removes the pid, and (4) removes the instance lockfile. Wired into `/api/shutdown` (replaces the existing 500ms `setTimeout` race), `process.on('SIGINT'|'SIGTERM')` (returns a Promise; second signal escalates to `exit(1)` so a hung close can't trap the user), and Tauri close (already routes through `/api/shutdown`).

**Live-cluster CHECKPOINT cadence.** Decision = no extra timer. The 5-minute backup tier already runs `CHECKPOINT` before each `dumpDataDir` (HS-7891), so the worst-case crash data-loss window is the same as the user's existing backup exposure. Adding a separate 60s timer would write to disk on idle machines for no benefit.

**Out-of-scope spikes** filed alongside the implementation: HS-7932 verifies PGLite's `fsync` round-trips through to host fs on each platform (concern: WASM ‚Üî host-fs bridge could silently no-op `fsyncSync`); HS-7933 benchmarks tighter `checkpoint_timeout = 60s` + `max_wal_size = 256 MB` to see if a smaller crash-recovery window is worth the disk-write delta.

**Status:** Mostly shipped. HS-7931 (graceful close) + HS-7934 (lock-release fold-in + e2e harness + signal-escalation unit test) + HS-7935 (explicit fsync wraps) all live. HS-7933 (checkpoint-timeout benchmark) blocked on PGLite upstream ‚Äî see HS-7936.

**Shipped (HS-7931 + HS-7934).** New `src/lifecycle.ts` exports `gracefulShutdown(reason)` + `registerHttpServerForShutdown(server)`. Pipeline: close HTTP ‚Üí destroy PTYs ‚Üí `closeAllDatabases()` (new export in `src/db/connection.ts`) ‚Üí `releaseAllLocks()` (HS-7934 fold-in; was previously a separate SIGINT handler in `src/lock.ts` that raced the async pipeline) ‚Üí remove instance file. Idempotent (concurrent callers share the same promise). Wired into `/api/shutdown` (replaces the previous 500ms `setTimeout(process.exit)` race) and the SIGINT/SIGTERM handlers in `cli.ts` via a new pure-factory `createSignalHandler(hooks)` (HS-7934 ‚Äî unit-testable without spawning a child process). A second identical signal during the await forces `exit(1)` so a hung close can't trap the user. Synchronous `process.on('exit')` stays as the safety-net lockfile remover. Tests: `src/lifecycle.test.ts` (8), `src/db/connection.test.ts` "closeAllDatabases" block (2), `src/cli.signalEscalation.test.ts` (5 ‚Äî pure escalation logic), `src/lifecycle.e2e.test.ts` (3 e2e scenarios + 1 skipped ‚Äî see HS-7939).

## 51. Per-terminal shell history scoping (`51-shell-history.md`)

HS-7964 (investigation) / HS-7965 (implementation). Each Hot Sheet terminal gets its own up-arrow history pool, scoped per (project, terminal id), so commands typed in project A's `zsh` terminal don't bleed into project B's `zsh` terminal ‚Äî and the Claude tab's nothing isn't mixed with the shell tab's recall.

**Why the simple `HISTFILE` env-var fix isn't enough.** Verified empirically: setting `HISTFILE` in the PTY's spawn env is silently clobbered on default-macOS zsh because `/etc/zshrc_Apple_Terminal` rewrites `HISTFILE` to a per-session file, and that rc runs AFTER our env injection but BEFORE the user's interactive prompt. So the fix has to inject AFTER the user's rc loads, which means using each shell's init-file override hook.

**Per-shell mechanism.** bash uses `--rcfile <path>` CLI rewrite (env vars don't work ‚Äî bash interactive shells only honour `--rcfile` for non-login interactive). zsh uses `ZDOTDIR=<dir>` env (cleanest). fish uses `XDG_CONFIG_HOME=<dir>` env + indexes by session NAME via `set -x fish_history hotsheet_<projectHash>_<terminalId>`. Other shells (sh / dash / ksh / pwsh / cmd) skipped ‚Äî no equivalent override worth implementing.

**Storage.** `<dataDir>/.hotsheet/shell_init/<terminalId>/` for generated rc files, `<dataDir>/.hotsheet/shell_history/<terminalId>` for bash + zsh history files. Fish history lands wherever fish puts it under the redirected `XDG_CONFIG_HOME` (session-name keyed by 8-char SHA-256-hex of dataDir + terminalId so multi-project doesn't collide). All under `<dataDir>/.hotsheet/` which is gitignored.

**Wire-up.** New `src/terminals/shellHistory.ts` exposes `setupShellHistoryForSpawn({dataDir, terminalId, command}): {env, rewrittenCommand, shell}` + 9 pure helpers. `src/terminals/registry.ts::spawnIntoSession` calls it after `resolveTerminalCommand`; the PTY spawn uses `command: rewrittenCommand ?? resolved.command` (bash gets `--rcfile` injection; others unchanged) and `env: buildEnv(shellInit.env)` (existing scrub + standard vars + per-shell override vars). `buildEnv` extended from no-arg ‚Üí `buildEnv(extra: Record<string, string> = {})`.

**Settings.** New per-project key `terminal_history_scope: 'per-terminal' | 'inherit'` (default `'per-terminal'`). `'inherit'` falls back to pre-fix behaviour (no override). No UI in v1 ‚Äî power users edit settings.json by hand.

## 61. Composable testable stores design (`61-composable-stores.md`)

HS-8167. Layered convention on top of ¬ß60's signals primitive. Hard dependency on HS-8166 landing before any phase here can start.

**Problem.** Client state lives in three overlapping shapes: module-level globals in `src/client/state.tsx`, per-feature module-level maps (`dashboardState` / `drawerGridState` / `terminalState` after HS-8189 / HS-8222 / HS-8223 / HS-8224 bundled the per-file `let`s), and ad-hoc `subscribeTo*` helpers. Tests can't seam in cleanly, derivations are hand-coded, and lifetime / project-switch reset is per-feature.

**`defineStore({initial, actions})` factory.** Returns `{ state: ReadonlySignal<T>, actions, reset() }`. Three rules: state is read-only (consumers subscribe via `effect()`), actions are the only mutation surface, `reset()` always returns to `initial()`. A module-level registry tracks every store; new `resetAllStores()` hook fires on `switchProject` so the per-feature reset hand-wiring goes away. Convention: one consumer = signal, two+ consumers (or multi-step mutations, or cross-navigation lifetime) = store.

**Candidate stores in pain order.** `ticketsStore` (highest impact ‚Äî `loadTickets` / `setFilter` / `select` / `applyServerUpdate` / `removeTicket` / `optimisticUpdate` actions, derived `filtered` + `grouped` signals), `projectsStore`, `terminalsStore` (likely one store with derived per-surface views), `commandLogStore`, `channelStore` (consolidates state spread across `channelUI.tsx` + `permissionOverlay.tsx`).

**Migration plan.** Phase 1 (HS-8238) factory + project-tab attention-dot trial + written test pattern. Phase 2 (HS-8239 ‚Äî atomic) `ticketsStore` end-to-end. Phase 3 (HS-8240) opportunistic per-store sub-tickets (commandLog view-layer migration HS-8311 was deferred + bundled into HS-8240 since the view + state migrations interlock).

**Pre-implementation investigation rejected nanostores backbone (HS-8248).** Evaluated `nanostores` (~294 bytes core) as a drop-in for the bespoke `defineStore`. Rejected: nanostores' `.get()` doesn't auto-track via the `signal.value` getter protocol, so ¬ß60's `effect()`-based binding helpers can't consume nanostores without either parallel `bindTextNano` helpers OR a manual signal-mirror at every consumer ‚Äî both reintroduce the "two reactivity systems" smell ¬ß60 + ¬ß61 exist to remove. Also: `cleanStores` semantics tear subscribers down (wrong shape for `resetAllStores` reset-to-initial), and the actions-only wrapper would be the bespoke factory regardless. Rationale archived in ¬ß61.13.

**Implementation update (HS-8235 / HS-8238 ‚Äî 2026-05-09):** the local `defineStore` factory described in ¬ß61.3 was NOT implemented ‚Äî kerf ships the same shape verbatim (read-only `state.value`, actions-only mutation, always-defined `reset()`, module-level registry walked by `resetAllStores()`). `src/client/reactive.ts` re-exports `defineStore` / `resetAllStores` / `Store` from `kerfjs`. Production `resetAllStores()` wiring deferred ‚Äî every store landed so far holds CROSS-project state (attention dots) that shouldn't reset on switch; re-evaluate during HS-8239 (`ticketsStore` IS per-project and would be the natural first production caller).

## 62. Unified JSX render targets via shared AST design (`62-unified-jsx-render-targets.md`)

HS-8168. Independent of ¬ß60 and ¬ß61. Addresses the third "honest pain point" HS-8165 called out: server `SafeHtml` vs client `toElement(<jsx />)` divergence. Both emit ostensibly equivalent output, but `el.innerHTML = jsx.toString()` round-trips break subtly on SVG namespacing, attribute escape edge cases, custom attributes (`is`, `slot`, `xlink:href`, parser-quirky `<table>` / `<select>` content), and whitespace.

**Two options.** Option A ‚Äî regression-test harness against the existing dual-target architecture (~50-case corpus; incremental, low-risk, but doesn't actually fix the bug class ‚Äî only catches divergences we think to test for). Option B ‚Äî unified runtime emitting an internal `JsxNode` AST + two consumers: `astToHtml` (server, today's semantics) + `astToDom` (client, builds DOM directly via `createElementNS` for SVG, `setAttribute` for attrs, `appendChild` for children ‚Äî no `innerHTML` round-trip). **Original verdict: Option B.** Higher upfront cost (~1.5‚Äì2 days) but permanently closes the bug class, and the dual-consumer shape is the more interesting open-source artifact.

**Re-scope under HS-8315 (2026-05-09).** Post-kerf-adoption, the much simpler path is to route Hot Sheet's `toElement` through `kerfjs::toElement`, which already implements the SVG-namespace fix (the bug class ¬ß62 cared about) ‚Äî same `SVG_FRAGMENT_TAGS` set, same `DOMParser('image/svg+xml')` for SVG fragments, same `<template>.innerHTML` fallback for HTML ‚Äî byte-for-byte equivalent for HTML JSX. **Phase 1 (HS-8241) shipped 2026-05-09.** `src/client/dom.ts::toElement` is now a thin `(jsx) => kerfToElement(jsx.toString()) as HTMLElement` wrapper; the 244 existing call-sites are unchanged. 5 new SVG bug-class tests in `dom.test.tsx` (renamed from `.ts`). Phase 2 (HS-8242 ‚Äî original "flip toElement to astToDom" plan) merged into Phase 1 since the same swap fulfilled both. Phase 3 (HS-8243) reduced in scope to just the `no-restricted-syntax` lint rule against direct `innerHTML = ` assignments outside an allowlist (94 production callsites ‚Äî would be a multi-day refactor; held for later).

**AST shape.** `JsxNode { tag, props, children }` with `JsxChild = JsxNode | string | { __raw: string } | null | undefined | false | true`. Plain string children auto-escape; `raw(html)` produces `{__raw}` markers (the only `innerHTML` use in the new runtime, scoped to the explicit raw escape hatch). `JsxNode.toString()` calls `astToHtml(this)` so every existing server `SafeHtml.toString()` callsite is unchanged.

**Migration plan.** Phase 1 (HS-8241) lands the AST + `astToHtml` parity + `astToDom` skeleton + 50-case equivalence harness; `toElement` keeps using `innerHTML` so the new path is validated in isolation. Phase 2 (HS-8242) flips `toElement` to call `astToDom` directly when handed a `JsxNode`; full e2e + bundle-size gate. Phase 3 (HS-8243) wires the equivalence corpus into CI + lint rule warning on direct `innerHTML =` outside the explicit allowlist.

**What stays the same.** JSX surface (`<div>`), `SafeHtml` API, `raw(html)`, all call sites, server `pages.tsx`, the HS-6341 / HS-6342 'DOM-children-throw' defensive guard. Unchanged.

**Status:** Design only. Cross-refs: ¬ß60 / ¬ß61 (independent tracks ‚Äî no dependency either direction), HS-8165 (origin), HS-6341 / HS-6342 (defensive throw preserved), ¬ß17 of `docs/ai/code-summary.md` (JSX runtime + bundle entry).

## 56. Magnified terminal-grid navigation (`56-magnified-grid-nav.md`)

HS-8028. While a terminal tile is centered (single-click overlay) or dedicated (double-click full-pane) in the ¬ß25 dashboard or ¬ß36 drawer-grid, **Shift+Cmd+Arrow** (macOS) / **Shift+Ctrl+Arrow** (Linux/Windows) switches the magnified target to the next tile in the indicated direction.

**Direction algorithm.** Strict grid-neighbour. For `left` / `right` the candidate's bounding-rect must overlap with `from` vertically (same row) AND lie strictly in the indicated half-plane; for `up` / `down` same shape with horizontal overlap. Among qualifying candidates the smallest edge-gap wins. When no candidate qualifies, the chord is a no-op. Pre-fix used a perpendicular-weighted cone metric that would jump rows when no same-row neighbour existed; the HS-8028 follow-up reply pinned the corrected behaviour ("if i click left, it should show the next terminal that is immediately to the left of the current one ‚Äî ignored if none"). Pure geometry helper `pickGridNeighbourIndex` lives in `src/client/gridNavGeometry.ts` (15 unit tests).

**Modes.** Centered AND dedicated. Bare grid (no magnification) doesn't bind the listener ‚Äî there's no "current magnified tile" to navigate from. Dedicated path forces `priorCenteredTile = null` before exit so the swap doesn't flash through a stale centered state.

**Chord shape.** Shift layered on top of the platform primary modifier (Cmd on macOS, Ctrl on Linux/Windows). Stays out of the way of `isJumpShortcut` (Cmd/Ctrl+Up/Down for OSC 133 prompt-marker jumps within a single terminal ‚Äî HS-7269). Wrong-platform modifier passes through unchanged.

**Capture phase + xterm interaction.** Document-level keydown listener uses capture phase + `preventDefault` + `stopPropagation` so xterm doesn't translate the chord into shell escape sequences (xterm's default for Cmd/Shift+Arrow is `\e[1;9X`-style sequences).

**Status:** Shipped. ## 21. Feedback notes (`21-feedback.md`)

AI tools request user feedback by adding a note to a ticket whose text begins with a recognized prefix (checked only on the most recent note):

- `FEEDBACK NEEDED:` ‚Äî shows dialog when ticket is opened.
- `IMMEDIATE FEEDBACK NEEDED:` ‚Äî auto-selects the ticket in the project's tab and opens the dialog.

Dialog shows prompt as markdown, one or more textareas, file attachments, and three buttons: **Later** (muted link, dismiss), **No Response Needed** (adds `NO RESPONSE NEEDED` note, clears state), **Submit** (creates note with response, uploads attachments, notifies channel). Click-outside dismisses like Later. Dialog auto-shows **once per detail-panel open** (tracked by note ID). "Provide Feedback" link button appears below a feedback-prefix note if it's the most recent.

**HS-6998 click-to-insert inline responses** (¬ß21.2.1): earlier auto-split-by-list heuristic (first qualifying list ‚Üí per-item textareas) misread prompts where the list was an A/B/C/D options menu rather than independent questions. Current approach does NOT guess: the prompt is split into top-level markdown blocks (paragraphs / lists / headings / code) via `parseFeedbackBlocks` in `src/client/feedbackParser.ts`, each block renders as an accent-bordered pill, and between every pair of blocks (and after the last) a muted `+ Add response here` button lets the user insert an inline textarea at that position. Inline responses have a `√ó` remove button; multiple responses per slot are allowed. A catch-all textarea always sits at the bottom. Submit semantics: (a) no inline responses used ‚Üí catch-all text is the note body verbatim (no quoting); (b) any inline responses used ‚Üí `combineQuotedResponse(blocks, inlineResponses, catchAll)` re-emits the whole prompt as markdown blockquotes (`> ` prefix on every line of every block) with the user's inline responses interleaved un-quoted in their slots and the catch-all appended un-quoted at the end. Empty-whitespace inline responses are dropped rather than rendered as placeholders. Covered by unit tests in `feedbackParser.test.ts`.

**HS-7558 per-list-item insertion points** (¬ß21.2.2): the v1 always rendered a list as a single block, so a numbered list of questions only got ONE `+ Add response here` slot at the end of the whole list ‚Äî users wanted to answer question-by-question. New `shouldSplitListIntoItems(list, isFirstBlock)` heuristic in `parseFeedbackBlocks` splits a top-level list into one block per item when EITHER (a) the list is the first meaningful block of the prompt (the screenshot scenario ‚Äî leading numbered question lists are almost always question sets regardless of punctuation), OR (b) every item's first line ends with `.`, `?`, or `!`. Sub-bullets nested under a list item stay grouped with their parent so a "1. Top question? ‚Äî sub a ‚Äî sub b" block keeps the parent + clarifications together. `combineQuotedResponse` is unchanged ‚Äî `blockIndex` indexes into whatever block list the parser returns, so per-item blocks slot in transparently. 4 new unit tests in `feedbackParser.test.ts` cover the four cases (leading list always splits, mid-prompt sentence-punctuated list splits, mid-prompt option list stays grouped, partial-punctuation list stays grouped) plus a sub-bullet-preservation test.

**HS-7599 Save Draft + don't-close-on-clickaway** (¬ß21.2.3): the dialog now persists in-progress responses to a new `feedback_drafts` table when the user clicks the new Save Draft button, and click-outside-the-dialog dismissal is gated on whether ANY input has text (catch-all OR any inline textarea). Dialog gains an optional `draftSeed` parameter that bypasses `parseFeedbackBlocks` and uses the draft's saved blocks verbatim ‚Äî load-bearing because future changes to `parseFeedbackBlocks` heuristics shouldn't reshape existing drafts when re-opened. Each draft renders as a dashed-bordered card with a "Draft" badge in the notes list, inline below its FEEDBACK NEEDED parent (or free-floating at the end if the parent's been deleted ‚Äî drafts persist as free-floating per the ¬ß21.2.3 lifecycle rule). Click reopens the dialog with the saved partition structure restored verbatim; right-click ‚Üí Delete Draft context menu. Storage: new `feedback_drafts` table (`id TEXT PK`, `ticket_id INTEGER FK ON DELETE CASCADE`, `parent_note_id TEXT NULLABLE`, `prompt_text`, `partitions_json`, timestamps), local-only ‚Äî NOT replicated through plugin sync engines. API: `GET/POST/PATCH/DELETE /api/tickets/:id/feedback-drafts[/:draftId]`. Implementation: `src/db/feedbackDrafts.ts` (CRUD + `FeedbackDraft` shape + `SavedPartitions` schema), four routes in `src/routes/tickets.ts`, `FeedbackDraftCreate/UpdateSchema` in `src/routes/validation.ts`, table DDL in `src/db/connection.ts`. Client: `src/client/feedbackDialog.tsx` extends `showFeedbackDialog()` with optional `draftSeed` parameter + `parentNoteId` arg + Save Draft handler + the `overlayHasAnyText`/`collectPartitions`/`partitionsHaveText` helpers; `src/client/noteRenderer.tsx` exposes `setTicketDrafts(ticketId, drafts)` + `getTicketDrafts(ticketId)` accessors and `buildDraftEntry(ticketId, draft, notes)` for rendering draft cards inline; `src/client/detail.tsx` GETs `/feedback-drafts` on detail-panel-open and re-renders with drafts populated. Tests: 7 unit tests in `src/db/feedbackDrafts.test.ts` (CRUD round-trips, partition idempotency, listing order, free-floating drafts, missing-id PATCH returns null, FK CASCADE on parent ticket delete) + 4 e2e in `e2e/feedback-drafts.spec.ts` (empty-input click-away closes / populated-input click-away keeps open / Save Draft persists + renders inline / click-to-reopen restores catch-all + Submit deletes the draft).

**Indicators:**
- Ticket dot: **purple** (#8b5cf6) for pending feedback, takes priority over blue unread dot.
- Project tab: purple dot while any ticket has pending feedback; priority order feedback (purple) > permissions attention (blue) > channel busy (yellow).

**Channel notification:** on submit (when channel alive), sends `triggerChannelAndMarkBusy` with text "Feedback was provided on ticket {ticketNumber}. Please re-read the worklist and continue work on this ticket." The worklist.md includes a "Requesting User Feedback" section with curl examples. ## Tauri architecture & setup (`tauri-architecture.md`, `tauri-setup.md`)

**Both missing from CLAUDE.md reading order ‚Äî add them.**

**Architecture:** Tauri v2 wraps Node as a sidecar because PGLite (WASM) needs filesystem access ‚Äî single-binary compilers break it. Layout: `src-tauri/{src, tauri.conf.json, Cargo.toml, Entitlements.plist, capabilities, loading, resources, binaries, server, icons}`. Three launch flows: (1) double-click (no `--data-dir`) ‚Üí welcome.html ‚Üí CLI install wizard; (2) CLI launch on macOS (complex due to JIT/filesystem restrictions) ‚Üí CLI script resolves app name, creates a stub `.app` in `.hotsheet/`, starts Node in background, writes URL+PID to `/tmp/hotsheet-server-{hash}.info`, `open -a` with stub app ‚Üí stub launcher reads `/tmp`, sets `HOTSHEET_SERVER_URL`+`HOTSHEET_SIDECAR_PID`, execs real Tauri binary with `--data-dir`; (3) Tauri with `--data-dir` (fallback/dev) ‚Üí spawns sidecar, waits for "running at" on stdout. Sidecar lifecycle tracked via `SidecarPid` managed state, killed on `RunEvent::Exit` (Unix: `libc::kill(-pid, SIGTERM)`; Windows: `taskkill /T /F`). Build pipeline: `scripts/build-sidecar.sh` downloads Node v20, runs `npm run build`, copies `dist/` + assets into `src-tauri/server/`; `tauri build` produces `.app`/`.AppImage`/`.msi`. CI/CD (`release-desktop.yml`) builds 4 targets, signs macOS, signs updates, creates draft GitHub Release. macOS entitlements: `allow-jit`, `allow-unsigned-executable-memory`, `disable-library-validation` (required for signed V8/WASM).

**Setup:** Prerequisites = Rust + npm. One-time config: generate updater keys via `npx tauri signer generate -w ~/.tauri/hotsheet.key`, copy pubkey to `tauri.conf.json plugins.updater.pubkey`; generate icons via `npx tauri icon path/to/1024.png`; set GitHub secrets `TAURI_SIGNING_PRIVATE_KEY` + password. macOS code-signing needs Developer ID Application cert + app-specific Apple ID password (secrets: APPLE_CERTIFICATE, APPLE_CERTIFICATE_PASSWORD, APPLE_SIGNING_IDENTITY, APPLE_ID, APPLE_PASSWORD, APPLE_TEAM_ID). Release: tag `v*` ‚Üí CI builds/signs/publishes draft release. Version must stay in sync across `package.json`, `src-tauri/tauri.conf.json`, and `src-tauri/Cargo.toml`.

**Status:** Shipped. Windows signing intentionally skipped.
